# Microsoft Teams「何もしなくても重い」問題、ついに修正へ

## ms-teams_modulehost.exe による通話スタック分離アーキテクチャの技術解説

---

**公開日**: 2025年12月  
**対象読者**: ソフトウェアエンジニア、インフラエンジニア、IT管理者  
**キーワード**: Microsoft Teams, WebView2, プロセス分離, パフォーマンス最適化, Chromium

---

## 概要

Microsoft Teamsは、リモートワークの普及とともに企業のコミュニケーション基盤として不可欠な存在となっている。しかし、Windows版Teamsは長年にわたり「起動が遅い」「会議中にフリーズする」「何もしていなくてもメモリを大量消費する」といったパフォーマンス問題が指摘されてきた。

2025年11月25日、MicrosoftはMicrosoft 365 Admin Center（Message Center ID: MC1189656）を通じて、この問題に対する構造的な改善策を発表した。**2026年1月より、Windows版Teams デスクトップクライアントに新しい子プロセス `ms-teams_modulehost.exe` が導入され、通話機能が本体アプリケーションから分離される。**

本記事では、このアーキテクチャ変更の技術的背景、実装の詳細、IT管理者が準備すべき事項、そして長期的な展望について解説する。

---

## 1. 現行Teamsアーキテクチャの課題

### 1.1 WebView2ベースアーキテクチャの特性

現在のMicrosoft Teams（いわゆる「新しいTeams」）は、2023年にElectronベースから**WebView2**ベースへ移行した。WebView2は、Microsoft Edge（Chromium）のレンダリングエンジンを利用してWebアプリケーションをデスクトップアプリとして動作させるフレームワークである。

```
┌─────────────────────────────────────────────────────┐
│                   ms-teams.exe                      │
│  ┌───────────────────────────────────────────────┐  │
│  │              WebView2 Runtime                 │  │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────────────┐ │  │
│  │  │ Browser │ │Renderer │ │  GPU Process    │ │  │
│  │  │ Process │ │Process  │ │                 │ │  │
│  │  └─────────┘ └─────────┘ └─────────────────┘ │  │
│  │  ┌─────────────────────────────────────────┐ │  │
│  │  │       Calling Stack (Audio/Video)       │ │  │
│  │  │  - コーデック処理                        │ │  │
│  │  │  - デバイスハンドリング                  │ │  │
│  │  │  - ネットワークネゴシエーション           │ │  │
│  │  └─────────────────────────────────────────┘ │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
             【従来のアーキテクチャ】
```

**WebView2採用の利点:**

- クロスプラットフォーム対応（Web/デスクトップで共通コードベース）
- 機能追加が容易（Web-firstの開発モデル）
- Edgeランタイムとの共有によるリソース重複の削減
- アプリ連携エコシステムの統一

**WebView2の本質的な課題:**

- Chromiumのマルチプロセスモデルを継承
- メモリを積極的にキャッシュとして確保する設計思想
- レンダリング最適化のため、メモリ解放より保持を優先

### 1.2 パフォーマンス問題の実態

ユーザーから報告されている主な症状:

| 症状 | 詳細 |
|------|------|
| **起動時間の長さ** | コールドスタートで10秒以上かかるケースも |
| **アイドル時のメモリ消費** | チャットのみでもRAM 500MB〜1GB超 |
| **会議中のフリーズ** | カメラ/マイクドライバとの競合でUI全体が停止 |
| **クラッシュの連鎖** | メディア処理の障害がアプリ全体を巻き込む |
| **システム全体への影響** | 他アプリケーションの応答性低下 |

これらの問題は、**単一プロセス内でUI処理と重いメディア処理が混在している**ことに起因する。音声・映像処理はCPU/I/O集約型であり、コーデックのバインディングやドライバとのやり取りで予期しないブロッキングが発生すると、UIスレッドまで影響を受ける。

## 2. 新アーキテクチャ: ms-teams_modulehost.exe

### 2.1 プロセス分離の設計

2026年1月のアップデートで導入される新アーキテクチャでは、通話スタック（Calling Stack）が独立した子プロセスとして分離される。

```
┌─────────────────────────────────────────────────────┐
│                   ms-teams.exe                      │
│  ┌───────────────────────────────────────────────┐  │
│  │              WebView2 Runtime                 │  │
│  │  - UIレンダリング                              │  │
│  │  - チャット機能                               │  │
│  │  - ファイル共有                               │  │
│  │  - Teams Apps                                 │  │
│  └───────────────────────────────────────────────┘  │
│                        │                            │
│                        │ IPC                        │
│                        ▼                            │
│  ┌───────────────────────────────────────────────┐  │
│  │         ms-teams_modulehost.exe               │  │
│  │  (Child Process)                              │  │
│  │  ┌─────────────────────────────────────────┐  │  │
│  │  │           Calling Stack                 │  │  │
│  │  │  - 音声処理 (Audio Pipeline)            │  │  │
│  │  │  - 映像処理 (Video Pipeline)            │  │  │
│  │  │  - デバイスハンドリング                 │  │  │
│  │  │  - ネットワークネゴシエーション          │  │  │
│  │  │  - メディアコーデック                   │  │  │
│  │  └─────────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
              【新アーキテクチャ】
```

### 2.2 期待される改善効果

#### 1. 障害分離 (Fault Isolation)

```
従来: カメラドライバ障害 → 通話スタッククラッシュ → ms-teams.exe全体クラッシュ
新規: カメラドライバ障害 → modulehost.exeクラッシュ → 再起動で復旧、UIは継続動作
```

メディア処理で障害が発生しても、チャットやファイル共有などの機能は継続して利用可能となる。

#### 2. 起動時間の短縮

通話機能の初期化（コーデック、デバイス列挙、メディアパイプライン構築）を別プロセスで並行実行することで、ユーザーがUIにアクセスできるまでの時間を短縮する。

#### 3. リソース競合の軽減

重いメディア処理がメインプロセスのUIスレッドと競合しなくなるため、会議中でもチャット入力やファイル操作の応答性が改善される。

#### 4. QoS (Quality of Service) の精度向上

ネットワーク管理者は、音声・映像トラフィックを生成する実際のプロセス（modulehost.exe）に対して、より正確なトラフィックマーキングやスケジューリングを適用できる。

### 2.3 技術的な制限事項

この変更は**WebView2アーキテクチャを維持したままの最適化**であり、以下の点は変わらない:

- Chromiumベースのマルチプロセスモデルとメモリ特性
- JavaScript/HTML UIレイヤーの構造
- クロスプラットフォームコードベース

つまり、WebView2に起因する基本的なメモリ消費パターンは解決されない。Microsoftの公式発表でも具体的なパフォーマンス数値は公開されておらず、実際の効果は環境依存となる。

## 4. IT管理者向け準備事項

### 4.1 セキュリティソフトウェアの設定

**最優先タスク: 許可リストへの追加**

新しい実行ファイル `ms-teams_modulehost.exe` を以下のセキュリティ製品で許可する必要がある:

- エンドポイント保護 (EPP/EDR)
- アプリケーション制御 (AppLocker, WDAC)
- ファイアウォール
- ウイルス対策ソフト

**許可設定を怠った場合のリスク:**

- チャット機能は動作するが、通話が開始できない
- 原因特定が困難な断続的な障害
- ヘルプデスクへの問い合わせ増加

### 4.2 QoSポリシーの更新

既存の `ms-teams.exe` に適用していたQoSポリシー（DSCPマーキング等）を `ms-teams_modulehost.exe` にも適用する。

**Windowsグループポリシー設定例:**

```
コンピューターの構成
└── ポリシー
    └── Windowsの設定
        └── ポリシーベースのQoS
            └── [新規ポリシー]
                アプリケーション名: ms-teams_modulehost.exe
                DSCP値: 46 (EF - 音声用)
```

### 4.3 ヘルプデスクへの周知

サポートスタッフに以下の情報を共有:

1. 新しいプロセスは正常な動作である
2. タスクマネージャーに見慣れないプロセスが表示されても、マルウェアではない
3. トラブルシューティング時は両プロセスの状態を確認する

### 4.4 ドキュメントの更新

内部ドキュメントでTeamsのプロセス名を参照している場合、`ms-teams_modulehost.exe` を追記する。

### 4.5 監視・テレメトリの設定

本格展開前のパイロットテストで以下のメトリクスを収集することを推奨:

```
監視対象メトリクス:
├── プロセス別CPU使用率
├── プロセス別メモリ使用量
├── クラッシュ頻度（ms-teams.exe vs modulehost.exe）
├── 通話開始までの時間
├── ネットワークカウンター
└── ユーザー報告の症状分類
```

## 5. 技術的考察: 根本解決か延命策か

### 5.1 プロセス分離パターンの妥当性

プロセス分離によるパフォーマンス改善は、ソフトウェアアーキテクチャにおいて確立されたパターンである:

- **Chromeブラウザ**: タブごとにプロセス分離（サイト分離）
- **Windows 11 File Explorer**: シェル拡張の分離による安定性向上
- **VS Code**: 拡張機能ホストプロセスの分離

Teamsの変更もこの文脈では合理的であり、特にリアルタイムメディア処理のような障害リスクの高いコンポーネントを分離することは有効なアプローチである。

### 5.2 WebView2維持の理由

一部のエンジニアからは「ネイティブアプリ（UWP/WinUI）として再構築すべき」との意見がある。しかし、Microsoftがこの選択を取らない理由は明確である:

| 観点 | WebView2維持 | ネイティブ再構築 |
|------|--------------|------------------|
| 開発コスト | 既存コードベース活用 | 大規模な再開発が必要 |
| クロスプラットフォーム | Web/macOS/Linux共通 | プラットフォーム別実装 |
| 機能追加速度 | Web-first開発で高速 | プラットフォーム別対応必要 |
| Teams Apps互換性 | 維持 | 再検証必要 |
| 技術的負債 | 継続 | リセット |

コストと機能速度のトレードオフにおいて、Microsoftはインクリメンタルな最適化を選択した。

### 5.3 類似事例: Meta WhatsApp Desktop

興味深いことに、MetaのWhatsAppデスクトップ版も同様にWebView2ベースアーキテクチャを採用しており、同様のパフォーマンス問題が報告されている。これは、WebView2が持つ本質的な特性であり、Teams固有の問題ではないことを示している。

## 6. 今後の展望

### 6.1 短期的な期待

ms-teams_modulehost.exe導入により、以下の改善が期待される:

- 会議参加までの体感時間短縮
- 会議中のクラッシュ頻度低下
- クラッシュ発生時の影響範囲限定

ただし、定量的な効果はハードウェア、ドライバ、ネットワーク環境に依存する。

### 6.2 中長期的な課題

根本的なパフォーマンス改善には、以下の取り組みが必要となる可能性がある:

- ランタイムメモリ管理の最適化
- スマートなアプリサスペンション機構
- 将来的なアーキテクチャ再設計（ネイティブコンポーネントの増加）

### 6.3 管理者へのアドバイス

1. **プロアクティブな準備**: 1月のロールアウト前に許可リストとQoSポリシーを更新
2. **パイロットテストの実施**: 会議負荷の高いユーザーと低スペックマシンでの検証
3. **期待値の管理**: 「魔法のような改善」ではなく「ターゲットを絞った最適化」と認識
4. **継続的な監視**: デプロイ後のメトリクス収集と問題の早期発見

## まとめ

Microsoft Teamsへの `ms-teams_modulehost.exe` 導入は、長年のパフォーマンス問題に対する**プラグマティックなエンジニアリング対応**である。プロセス分離により通話機能の安定性と起動時間の改善が期待される一方、WebView2アーキテクチャに起因する基本的なリソース消費パターンは維持される。

IT管理者は、セキュリティポリシーの更新、QoS設定の適用、ヘルプデスクへの周知を事前に行い、スムーズな移行を実現することが求められる。

この変更は「延命策」と「根本解決」の間に位置するが、Microsoftのソフトウェア開発戦略（クロスプラットフォーム・Web-first）を考慮すれば、現時点で最も現実的なアプローチと言える。
